# План: Універсальна система варіантів товарів

## Суть концепту

Один товар може мати **довільний набір груп опцій** (наприклад: Обʼєм памʼяті, Колір, Стан, Підключення — все залежить від типу товару). Кожна унікальна комбінація опцій — це **варіант** зі своєю URL, ціною, залишком і SEO-метаданими.

```
Товар: "Samsung T7 Shield"
  ├── Група: Обʼєм памʼяті   → [1TB, 2TB]
  ├── Група: Колір            → [Бежевий, Синій, Чорний]
  └── Група: Стан             → [Новий, Б/У]

Варіанти (кожен — окрема сторінка):
  /product/samsung-t7-shield-1tb-beige-new
  /product/samsung-t7-shield-1tb-beige-used
  /product/samsung-t7-shield-2tb-black-new
  ...і так далі
```

Це працює для **будь-якого** типу техніки — телефонів, ноутбуків, карт памʼяті, навушників, дронів тощо. Адмін не налаштовує "поля для телефонів" — він просто додає групи з будь-якими назвами.

---

## Поточний стан (що вже є)

| Компонент | Стан | Деталі |
|---|---|---|
| Prisma модель `Variant` з унікальним `slug` | ✅ Є | Кожен варіант вже має поле для своєї URL |
| `ProductOptionGroup` / `ProductOptionValue` | ✅ Є | Довільні групи і значення, привʼязані до конкретного товару |
| `VariantOptionSelection` (junction table) | ✅ Є | Звʼязок варіант ↔ значення опції |
| `findBySlug()` — шукає і по Product.slug і по Variant.slug | ✅ Є | Основа для SEO-роутингу варіантів |
| `ProductDetail.tsx` — перемикач варіантів з оновленням URL | ✅ Є | UI вже змінює URL при виборі варіанту |
| `ProductForm.tsx` — генерація матриці варіантів в адмінці | ✅ Є | Є, але складно зрозуміти |
| Фільтрація каталогу по опціях варіантів | ❌ Немає | Каталог не вміє фільтрувати по групах |
| SEO metadata на рівні варіанту | ❌ Немає | Title/description однакові для всіх варіантів |
| Canonical URL для варіант-сторінок | ❌ Немає | Ризик дублікатів контенту для Google |
| `availableFilters` в API відповіді каталогу | ❌ Немає | Немає звідки брати список фільтрів для UI |
| Картка товару в каталозі веде на варіант | ❌ Немає | Посилання веде на базовий slug, не варіант |

**Висновок:** архітектура даних готова на ~60%. Потрібно добудувати шар SEO і фільтрації.

## Що треба зробити (6 блоків)

---

### Блок 1 — Бекенд: фільтрація каталогу по варіантах

**Файли:**
- `backend/src/products/products.service.ts` → метод `findAll()`
- `backend/src/products/dto/product.dto.ts` → `GetProductsQueryDto`

**Задача:**

Додати в query параметри можливість передавати опції варіантів:
```
GET /api/products?categorySlug=storage&options[Обʼєм памʼяті]=1TB&options[Стан]=Новий
```

Логіка фільтрації:
- Товар проходить фільтр якщо у нього є **хоча б один варіант**, який задовольняє **всі** передані групи одночасно
- JOIN: `Product → Variant → VariantOptionSelection → ProductOptionValue → ProductOptionGroup`

Додати в відповідь `findAll()` поле `availableFilters`:
```json
{
  "items": [...],
  "total": 42,
  "availableFilters": [
    {
      "groupName": "Обʼєм памʼяті",
      "values": ["256GB", "512GB", "1TB", "2TB"]
    },
    {
      "groupName": "Стан",
      "values": ["Новий", "Б/У"]
    }
  ]
}
```
Фільтри генеруються **динамічно** — з реальних варіантів товарів у поточній категорії з поточними іншими фільтрами. Не хардкодяться.

---

### Блок 2 — Бекенд: variant-level SEO metadata

**Файли:**
- `backend/src/products/products.service.ts` → метод `findBySlug()`

**Задача:**

Коли запит приходить на Variant.slug, повертати додатково:
```json
{
  "product": { ... },
  "selectedVariantId": "uuid-тут",
  "variantSeo": {
    "title": "Samsung T7 Shield 1TB Новий — купити в Smartline",
    "description": "Samsung T7 Shield 1TB Новий. Швидка доставка...",
    "canonicalUrl": "/product/samsung-t7-shield-1tb-new"
  }
}
```

Формула для `variantSeo.title`:
```
{Назва товару} {Значення опцій через пробіл} — купити в Smartline
```
Значення опцій беруться з `VariantOptionSelection` → `ProductOptionValue.value`, відсортовані по `ProductOptionGroup.sortOrder`.

---

### Блок 3 — Фронтенд: фільтри в каталозі

**Файли:**
- `frontend/app/(store)/catalog/[categorySlug]/page.tsx` (або поточний каталог)
- Новий компонент: `frontend/components/store/CatalogFilters.tsx`

**Задача:**

Server Component читає `availableFilters` з API → рендерить сайдбар або топ-бар з фільтрами.

Поведінка:
- Кожна група опцій — окрема секція з чекбоксами
- Вибір фільтру → `router.push()` з оновленими URL params
- URL виглядає: `/catalog/storage?memory=1TB&condition=new`
- При зміні фільтрів — skeleton loading
- На мобільному — фільтри в drawer/modal

Стилі: дизайн-токени `--sl-*`, темна тема, hover через `.sl-hover-*` CSS класи (не `onMouseEnter`).

---

### Блок 4 — Фронтенд: SEO для сторінок варіантів

**Файли:**
- `frontend/app/(store)/product/[slug]/page.tsx`

**Задача:**

1. `generateMetadata()` використовує `variantSeo.title` / `variantSeo.description` якщо є `selectedVariantId`
2. Додати `<link rel="canonical" href={variantSeo.canonicalUrl} />` в `<head>`
3. JSON-LD `Product` schema: ціна береться з обраного варіанту, а не з `basePrice`
4. OpenGraph image: перше зображення товару (або варіанту якщо буде окремо)

---

### Блок 5 — Фронтенд: картка товару в каталозі

**Файли:**
- `frontend/components/store/ProductCard.tsx`

**Задача:**

- Посилання на картці веде на `slug` **першого варіанту в наявності** (stock > 0), а не на базовий product slug
- Ціна на картці = ціна цього варіанту
- Якщо є варіанти з різними цінами — показувати "від X грн"
- Бейдж "Б/У" якщо найдешевший варіант є б/у

Логіка вибору "першого варіанту":
1. Варіанти з `stock > 0`, відсортовані по `sortOrder` або ціні
2. Якщо всі без залишків — перший варіант взагалі

---

### Блок 6 — Адмінка: спрощення форми варіантів

**Файли:**
- `frontend/components/admin/ProductForm.tsx`

**Задача:**

Зробити секцію варіантів зрозумілішою без зміни логіки:

**Крок 1 — Групи опцій:**
```
+ Додати групу опцій
┌─────────────────────────────────────────┐
│ Назва групи: [Обʼєм памʼяті        ]   │
│ Значення:    [256GB] [512GB] [1TB] [+] │
└─────────────────────────────────────────┘
```

**Крок 2 — Варіанти (таблиця):**
```
[Генерувати всі комбінації]

Комбінація          Ціна    Порівн. ціна  Залишок  SKU       Активний
256GB / Новий       12 000  —             5        SKU-001   ✅
256GB / Б/У         9 500   12 000        2        SKU-002   ✅
512GB / Новий       15 000  —             3        SKU-003   ✅
```

Зміни:
- Inline editing прямо в таблиці (без popup)
- Кнопка "Генерувати всі комбінації" — створює матрицю з поточних груп
- Tooltip на заголовку секції: *"Кожен рядок — окрема URL для пошукових систем"*
- SEO slug показується і редагується при розгортанні рядка (accordion)
- Підсвітка рядків з залишком = 0

## Порядок реалізації

```
1. Блок 1 — findAll() з фільтрами + availableFilters    ← основа, решта залежить від цього
2. Блок 2 — findBySlug() variant SEO                   ← паралельно з блоком 1
3. Блок 3 — Каталог UI з фільтрами                     ← після блоку 1
4. Блок 4 — SEO metadata + canonical                   ← після блоку 2
5. Блок 5 — ProductCard variant-aware                  ← після блоків 1-2
6. Блок 6 — Адмінка форма                             ← незалежно, будь-коли
```

---

## Що НЕ треба робити

- Хардкодити назви груп (не "Колір", не "Памʼять" — все динамічно)
- Окрема таблиця в БД під кожен тип товару — модель вже достатньо гнучка
- Окремі маршрути типу `/product/[id]/[variantSlug]` — поточний `[slug]` вже обробляє обидва випадки
- Ламати поточну логіку `ProductDetail.tsx` — вона вже добре обробляє варіанти

## Технічні обмеження (памʼятка)

| Правило | Причина |
|---|---|
| Server Components не можуть мати `onMouseEnter` на `<Link>` | Next.js 16 build error — використовувати `.sl-hover-*` CSS класи |
| `useRef<number \| null>(null)` для таймерів | `@types/node` конфліктує з браузерним `setTimeout` |
| Всі запити через `lib/api.ts → apiFetch` | `credentials: 'include'` для cookie-auth |
| Tailwind v4: `bg-[var(--sl-bg-surface)]` синтаксис | Немає `tailwind.config.ts`, токени через CSS variables |
| Prisma тільки в бекенді | Архітектурна вимога проекту |
| Адмінка — hardcoded light-theme Tailwind класи | Не застосовувати `--sl-*` dark токени в `/admin` |
